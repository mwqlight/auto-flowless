# 流程功能增强
这是一个开源的流程平台，但是由于其目前功能还是比较简单，因此需要增强其功能。需要在了解现有功能的基础上进行增强，注意是增强，不是替换，也就是原来的基本功能是不丧失的，具体的要求如下：
## 增强的功能点如下
1. 将feature-run的分支先合并到当前feature-flow-inhance分支，在feature-run分支已经开发好了项目依赖本地环境启动的基本功能，也就是说合并后，当前分支也能功能正常启动前后端。
2. 去掉水印功能
3. 在当前的设计流程图的时候，只能绘制比较基础的，比如会签或签等，还不支持触发器，监听器，延时器，定时器的功能，需要结合前后端的功能进行扩展，这里需要特别注意的是，这个触发器，监听器，延时器，定时器的高级功能是需要通过动态引入jar包的形式进行动态扩展的，而不是写死的，因此也需要有个菜单功能需要进行各种类型jar包的上传和管理。然后启用或者禁止，是需要在绘制流程的时候，可以选择不同类型的高级功能。当然这里还是比较模糊，因此需要明确区别场景，因为所有的高级功能的应用都是基于事件来生效的，因此配置高级功能，就不能像绘制流程那样在流程中进行配置，而是一个单独个高级功能，需要在绘制完流程后，那么就可以选取一个节点，或者一段节点，来抽取事件，从而配置是在那个事件上生效，而节点的事件比较单一，无非就是进入节点，离开节点，但是对于一段节点，就是指明开始节点和结束节点，则以为着是以连串的节点，同样可以理解为进入节点和离开节点，不过对象就分为两个，除了这些还有其他基本事件，流程开始，结束，挂起，解挂等，需要综合业务场景，开发一个完成的高级功能，涉及菜单，前后端的功能。
4. 在绘制流程的时候，支持的高级功能是一个在流程之上的功能，还需要在流程绘制的时候，能够加入一些内置的高级功能，比如触发器，监听器，延时器，定时器等，这些内置的功能和高级功能最大的区别在于是逻辑写死的，不需要jar包的，其内置的业务逻辑，可以是邮件通知，短信通知，信息通告等，可以自行发挥，将日常的功能集成内置的高级功能中去，还有就是绘制的功能元素和当前绘制流程的是一样的，就是右键可以选择对应的高级功能，然后生成相应的节点，当前高级功能是需要配置事件的，需要沿用当前的配置页面的形式，进行开发。
5. 在绘制流程的时候，需要新增的一个AI助手的选项，也作为节点最终在流程中显示，这个AI助手的节点的功能点有什么用了，就是可以配置选取的模型，当前就需要加入一个新的菜单用来管理当前项目集成了哪些模型，然后配置选取，需要注意的是选取的模型的业务功能是可以配置的，用来操作整个流程的表单内容，在执行流程的时候，当进入AI助手这个节点时候，就把当前的表单作为输入，然后在配置AI助手的节点中用语言描述，需要怎么操作表单。简单来说，就是系统集成各种AI大模型，然后在执行流程的过程中，可以通过执行流程来执行大模型，并且影响表单结果。
6. 删除auto-flowless-backend中imgs下的目录以及文件，最后开发完，因为已经合并了feature-run的功能，因此是可以正常启动的，所有最后需要正确启动前后端，进行功能验证，确保没有错误和警告。

## 开发注意事项
1. 需要有全局意识，当前的项目本身就是一个功能完整正确的项目，在开发新功能的时候，需要注意不要影响到原来的功能，也不要破坏原来的功能。
2. 启动的前后端的端口不能变，不能变，防止端口浪费。
3. 复杂的业务功能是需要涉及前后端整体开发的，需要有数据库的设计，和业务逻辑的整合。不能简单的页面就可以，是远远不够的。
4. 不能伪造，不能模拟，不能虚假。
5. 要注意，在开发复杂功能的时候，先进行原理分析，然后制定具体的实现方案，不能直接开始编码，否则会影响到原来的功能，也会影响到其他功能的开发。
6. 在完成本次任务的时候，先制定详细的实现方案，包括数据库的设计，业务逻辑的整合，前后端的功能实现等，然后再开始编码。要注意任务点之间的先后逻辑顺序，不能乱序，否则会影响到功能的实现。
